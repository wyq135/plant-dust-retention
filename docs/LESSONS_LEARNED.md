# 中文学术文献抓取：技术选型经验教训

> 2026-06-14 | 植物滞尘项目

## 问题起源

需要从18篇参考文献的期刊页面上核实论文元数据（标题、作者、摘要、环境数据），发现 Claude Code 内置的 WebFetch 工具无法访问绝大多数中文学术网站。

## 三条技术路线的试验结果

### 路线1: Claude Code WebFetch（内置浏览器工具）

| 域名 | 结果 | 原因 |
|------|:--:|------|
| `doi.org` | ❌ 拒绝 | 服务端安全域名白名单过滤 |
| `ecologica.cn` | ❌ 拒绝 | 同上 |
| `*.cn` 学术域名 | ❌ 全部拒绝 | Claude 服务端拦截所有 `.cn` 域名 |
| `skills.sh` | ❌ 拒绝 | 甚至技能商店也被拦 |

**结论**: WebFetch 的安全过滤器在 Anthropic 服务端，与本地网络/VPN 无关。用户无法配置绕过。

**适用场景**: 仅限 `.com`/`.org` 等已被 Anthropic 列入白名单的国际域名。

---

### 路线2: Firecrawl（第三方抓取 API）

- **安装**: `npx skills add firecrawl/firecrawl-workflows@firecrawl-research-papers`
- **API Key**: 免费版 1,000 credits/月，本项目最多用 ~350
- **实际测试结果**:

| 目标 | 结果 | 原因 |
|------|:--:|------|
| `ecologica.cn` (HTTPS) | ❌ 被劫持 | 网站被挂马，检测到浏览器 User-Agent 后注入色情广告 JS |
| `ecologica.cn` (HTTP) | ❌ 同上 | Firecrawl 使用真实浏览器渲染，JS 劫持生效 |
| `pjoes.com` (国际) | ⚠️ 部分可用 | 提取到内容但结构化差 |
| `sciencedirect.com` | ❌ 空内容 | Elsevier 付费墙 |

**结论**: Firecrawl 的价值在于国际期刊和 DOI 元数据查询。中文学术网站因大面积被黑/挂马，浏览器渲染路径不适用。

**适用场景**: 国际出版商网站、arXiv、PubMed 等没有恶意 JS 的学术平台。

---

### 路线3: curl + Python HTML 解析 ✅ 最终方案

**工作原理**:
```
DOI → curl -sL (跟随重定向) → 自动编码检测 → 正则清洗 → 结构化 JSON
```

**关键发现**:

1. **中文学术 DOI 的解析路径**:
   - `doi.org` → `chndoi.org`（中文 DOI 注册中心）→ 显示"多重解析地址选择页面"
   - 这个页面包含：题名、作者、DOI、出版商，但**不含**摘要和期刊名
   - 要进一步获取摘要/期刊名需要访问知网（CNKI），知网需要学校图书馆认证

2. **编码问题**:
   - 中文学术网站通常声明 `charset=utf-8` 但实际可能用 GBK/GB2312
   - 必须依次尝试 UTF-8 → GB18030 → GBK → GB2312 解码
   - Windows 终端 GBK 编码会乱码，但文件存储 UTF-8 没有问题

3. **不同期刊网站的可访问性**:

| 期刊平台 | curl 可访问 | 含摘要 | 含期刊名 | 备注 |
|----------|:--:|:--:|:--:|------|
| ecologica.cn | ✅ | ✅ | ✅ | 唯一完全成功的平台 |
| chndoi.org | ✅ | ❌ | ❌ | 仅标题+作者，期刊名被CNKI覆盖 |
| sciencedirect.com | ❌ | — | — | Elsevier 付费墙 |
| pjoes.com | ⚠️ | ⚠️ | ⚠️ | 国际期刊，HTML结构差异大 |

# v4 批量提取与仓库管理：经验教训

> 2026-06-23 | 植物滞尘项目 v4 第三批准备阶段

## 1. 英文论文搜索：paper-search CLI 不可靠

### 背景
需要为全国尺度数据集找英文论文补充。`paper-search` skill 覆盖22个学术源（Semantic Scholar/PubMed/Crossref等）。

### 实际结果
- Semantic Scholar API **连续429限流**，搜索全部返回0结果或500错误
- Semantic Scholar/Core/PubMed **对中国城市植物滞尘这一细分领域覆盖极差**——这是CNKI主导的领域
- 换用 **WebSearch**（搜 MDPI/Springer/ScienceDirect 站内）效果好得多，10分钟内找到8篇高质量论文含2篇系统综述

### 教训
- **paper-search CLI 适用于主流英文学术话题**（生物医学/CS/AI），不适合中国城市生态学细分领域
- **WebSearch 直接搜出版商站内**是更务实的替代方案
- 该领域目前是 **CNKI主导的学术生态**——Steinparzer(2023)系统综述中57篇论文51%来自中国，且多为中文期刊

## 2. WebFetch 全面被企业安全策略拦截

### 实际情况
- `sciencedirect.com` — "Unable to verify if domain is safe to fetch"
- `mdpi.com` — 同上
- 几乎所有学术出版商网站都被拦截

### 教训
- 在当前网络环境下，**WebFetch 基本不可用**
- 需要用外部工具（curl/浏览器/知网）获取全文
- 这进一步巩固了"用户下载PDF → Claude提取"的工作流

## 3. 文件分类：临时起意是万恶之源

### 问题
`downloaded/` 目录先后面临过：`_不清晰/`、`_纯图表跳过/`、`_图片表格/`、`_待人工识别/`等碎片目录。每次碰到新情况就新建一个目录，最终导致：
- 同一篇论文散落多处（李晨2013出现在3个目录）
- 不知道文件在哪里，需要全文检索
- 新会话的 agent 无法定位文件

### 解决方案
改为**基于处理生命周期的6状态互斥分类**：

```
_待处理/ → _待处理/_需VLM/ → _需重下/ → _待人工/ → _已入库/ → _已排除/
```

一篇论文在任何时刻只能处于一个状态。规则写入 `downloaded/README.txt` 和 CLAUDE.md。

### 教训
- **不要临时建目录**。先定义状态机，再分配文件。
- 分类必须是**互斥**的（一个文件不可能同时在两个目录）
- 规则必须**固化到文件系统本身**（README.txt），不能只存在 Claude 的会话上下文中

## 4. CLAUDE.md 数据描述会过时

### 问题
数据质量区块写着 `method_level: A=549 B=332 C=3` 和 `needs_manual_review: 74条`——这些都是 method_level 回填前的老数据，已严重过时（实际 A=635 B=1483 C=85, needs_manual_review=275）。

### 教训
- **统计数字必须能从数据集自动生成**，不能靠手动维护
- `_gen_stats_report.py` 已经能自动生成全部统计，**不再在 CLAUDE.md 中手写数字**
- 或者只在 CLAUDE.md 中保留概括性描述，具体数字引用自动生成的报告

## 5. "不确定就放_待人工，绝不放_已排除"

### 问题
如果一篇模糊PDF被误判为"无数据"放入 `_已排除/`，后续发现它其实有数据时，需要全文检索找回——浪费算力。

### 原则
> 确认无数据 → `_已排除/`（永不录用）
> 不确定 → `_待人工/`（你还会看）
> 这种保守原则确保：**宁可多看一篇，绝不漏掉一条数据**

## 6. 一次性脚本归档而非删除

### 做法
133个一次性脚本/中间产物/提取输出 → `Desktop/2026/_repo_archive/`

### 原因
- 这些脚本虽不再执行，但包含当时的提取逻辑、参数选择、错误处理——**它们是项目决策的可追溯记录**
- 归档到 Git 仓库外（避免污染仓库），但本地完整保留
- 几个月后如果需要查"张翼飞的mg/leaf当时是怎么转换的"，归档里有答案

## 7. Git 仓库的职责边界

### 原则
```
GitHub 公开仓库 = 文档 + 工具 + 轻量数据快照
永远不上传 = PDF（公开可下载）/ 临时脚本 / 中间产物 / 虚拟环境
```

### 实践
- `.gitignore` 排除 `*.pdf/*.png/*.bin`、`.venv/`、`output/`
- `tools/` 放可复用工具，`docs/` 放文档
- 133个临时脚本归档到 `Desktop/_repo_archive/`（不在Git中）

## 8. .bin CAJ文件是系统性瓶颈

26篇论文只有 `.bin` 格式（知网CAJ），pdfplumber无法读取。这些论文覆盖北京/福州/哈尔滨/杭州/昆明/南宁/厦门/湛江等多个城市。

**当前方案**：放入 `_需重下/`，等用户在知网重新下载PDF版本。

**长期**：如果能解决CAJ→PDF转换，这批论文可以释放。

## 9. 中文俗名→拉丁学名的权威来源

"小叶榕"在数据集中出现3个不同拉丁名（Ficus microcarpa/concinna/benjamina）。查《中国植物志》（FRPS）确认：小叶榕=雅榕=Ficus concinna，Ficus microcarpa实为榕树（不同物种），Ficus benjamina为垂叶榕（不同物种）。

**规范**：所有物种拉丁名以FRPS/Flora of China为权威来源，不以原始论文的野外鉴定为准。
